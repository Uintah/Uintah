Introduction

This section introduces the component specification document. You should have already read Module Design/Implementation Procedure

A postscript verison of this document can be found, relative to the top of your SCIRun CVS tree, in doc/ReferenceGuide/component.ps.


Purpose

Writing the XML component specification is one of the first steps in the design and implementation of a component.

The XML component specification documents the use, design, implementation, and testing of a SCIRun component.

It also serves as data for applications and databases. For example:

  • Documentation (html and pdf forms) for component users and developers will be generated from the component specification.

  • The collection of all component specifications will be part of a search-able database.

  • The component specification will help drive the component code generation process.

  • The component specification may drive an automated testing process.

Component specifications are written in a special mark-up language. This language (call it the component specification mark-up language if you like) is in turn formulated using the eXtensible Mark-up Language (XML).

Note: Components that are not accompanied by a valid and complete specification document will not be incorporated into official releases of SCIRun software.


Why XML?

An XML formulation of the component specification is used to achieve the following goals:

Data centric viewpoint
We want the component specification to represent a set of data that can be used in several contexts.

Uniformity
We want all components to be specified the same way.

Validation
We want to validate (check for correctness and completeness) component specifications.

The component specification mark-up language is similar in concept to HTML but has a different (and much smaller) set of tags and a simpler and more regular set of document composition rules. Section Rules, Structure, and Content describes the component specification mark-up language in detail.


The DTD

A Document Type Definition (DTD) defines a set of elements (and element attributes), and the way they must be used in order to produce a valid XML document.

The component specification DTD can be found here (relative to the top of your cvs tree): 

doc/ReferenceGuide/component.dtd

The component specification DTD is used to validate component specifications. More on validation can be found in section Validation.

The component specification DTD is used to create and edit component specifications. See section Editing the Spec for more information.


Rules, Structure, and Content

This section describes the component specification mark-up language in detail.


Rules

When writing a component specification a few simple composition rules must be followed:

  • In the XML world tags are used to delimit (or identify) elements. Elements are the basic building blocks of an XML document and contain content (text) or nested elements or both. Every start tag must have a corresponding end tag.

  • A component specification document starts with 2 lines of preamble text: 

    <?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE component SYSTEM 
    "<top-of-cvs-tree>/doc/ReferenceGuide/component.dtd}">

    where <top-of-cvs-tree> is the path to the top of your SCIRun CVS tree.

  • The preamble is followed by a set of nested elements starting with the <component> element.

  • Every element must be delimited with a start-tag and an end-tag. Tags are enclosed in angle brackets. This means you may not use the literal forms of the characters < and > in your elements' content. Instead you must type &lt; and &gt; respectively.

  • Tags are case sensitive and all tags must be lower case.

  • A few elements require an attribute. Attributes are specified in an element's start tag. An attribute looks like this: attribute_name="attribute_value". The attribute's value must be enclosed in double quotes. Attributes are documented below along side their corresponding element.

  • The literal form of the character & may not be used in element content. You must type &amp; instead.

The set of valid elements are described next.


Structure and Content

Below, the component specification's XML structure and corresponding logical content are described. Each element's start tag is followed by a description of its purpose and content, followed by its nested elements, and finally by its end tag. Indentation is used to show nesting relationships among elements. The rather complex <description> element, which is used in several contexts, is not described in detail here - section A Description of the <description> Element describes it fully.

<component name="component-name" category="category-name">
Starts the component specification. All other elements are nested within this one. This element requires the name and category attributes. These attributes name your component and specify the category the component belongs to respectively.
<overview>
Starts the overview section. Contains only nested elements.
<authors>
List of component's authors. Contains 1 or more <author> elements.
<author>
Name of 1 author.
</author>
</authors>
<summary>
Short (1 or 2 sentence) summary of component's function.
</summary>
<description>
Comprehensive description of component's function. The <description> element is a mini-documentation environment. See section A Description of the <description> Element for details on the use of the <description> element.
</description>
<examplesr>
The name of an .sr file that demonstrates the component's use.
</examplesr>
</overview>
<implementation>
Lists the names of a component's source code files excluding the component's primary source file (which is implied by the <component> element's name attribute).

The need for an implementation section is rare - the source code for most components is contained entirely within a component's primary source code file.

The subelements <ccfile>, <cfile>, and <ffile> list the names of c++, c, and fortran source code files respectively. These elements may occur multiple times and in any order. Note that file name extensions are omitted.

<ccfile>
Names 1 c++ file, e.g. <ccfile>foo</ccfile>.
</ccfile>
<cfile>
Names 1 c file.
</cfile>
<ffile>
Names 1 fortran file.
</ffile>
</implementation>
<io>
Component's inputs and outputs. Contains 2 elements, <inputs> followed by <outputs>.
<inputs lastportdynamic="(yes | no)">
Describes the component's inputs. Inputs may come from a port, a file, or a device. Zero or more <port> elements are used to describe inputs from ports. Zero or more <file> elements are used to describe inputs from files. Zero or more <device> elements are used to describe inputs from devices. There must be at least 1 <port> element or 1 <file> element or 1 <device> element.

If your component's last port is dynamic then the lastportdynamic attribute should be set to yes (lastportdynamic="yes"). If your component's last port is not dynamic then you may omit this attribute or set its value to no.

<port>
States that input data comes from a port. Sub-elements describe the data further.
<name>
The port's name.
</name>
<description>
High level description of the data accepted by the port, e.g. ``These are potential data from body surface of a human.''
</description>
<datatype>
The name of a SCIRun data type, i.e. the data type that is being transmitted on this port.
</datatype>
<componentname>
The name of an upstream component that is commonly used to send data to this component on this port. This element may occur multiple times.
</componentname>
</port>
<file>
States that input comes from a file. Sub-elements describe the data further.
<description>
High level description of the data provided by the file, e.g. ``These are potential data from body surface of a human.''
</description>
<datatype>
The name of a SCIRun data type, i.e. the data type being read from the file. Note: Only SCIRun data types may be read from files - no custom formats allowed!
</datatype>
</file>
<device>
States that input comes from a device. Sub-elements describe the data further.
<devicename>
Device's name.
</devicename>
<description>
High level description of the data provided by the device.
</description>
</device>
</inputs>
<outputs>
Describes the component's outputs. Outputs may go to a port or to a file or to a device. Use 0 or more <port> elements to describe outputs to ports, 0 or more <file> elements to describe outputs to files, and 0 or more <device> elements to describe outputs to devices.
<port>
States that output is sent a port. Sub-elements describe the data further.
<description>
A high level description of the data sent to the port, e.g. ``These are potential data from the body surface of a human.''
</description>
<datatype>
The name of a SCIRun data type, i.e. the data type that is being transmitted on this port.
</datatype>
<componentname>
The name of a downstream component that is commonly connected to this output port. This element may occur multiple times.
</componentname>
</port>
<file>
States that output is sent to a file. It's sub-elements describe the outputs.
<description>
A high level description of the data sent to the file, e.g. ``These are potential data from body surface of a human.''
</description>
<datatype>
The name of a SCIRun data type, i.e. the data type being written to the file.
</datatype>
</file>
<device>
States that output goes to a device. Sub-elements describe the output further.
<devicename>
Device's name.
</devicename>
<description>
High level description of the data sent to the device.
</description>
</device>
</outputs>
</io>
<gui>
States that the component supports a GUI. Sub-elements describe the GUI.
<description>
Describes the purpose of the GUI, e.g. ``The GUI allows you to steer the simulation.''
</description>
<parameter>
Declares 1 GUI parameter. A parameter is a GUI item that can be modified by the user. Sub-elements describe the parameter. Use 1 parameter element for each item in the GUI.
<widget>
The tcl widget used for representing this parameter.
</widget>
<datatype>
The type of data value stored for this parameter (e.g. int, double, bool, string).
</datatype>
<label>
Name of parameter as it appears in GUI.
</label>
<description>
Describes the purpose of the parameter and how it can be used to control the component's behavior.
</description>
</parameter>
<img>
Name of a file that contains a picture of the GUI.
</img>
</gui>
<testing>
Starts the testing section. Sub-elements specify 1 or more testing plans.
<plan>
Declares 1 testing plan. Sub-elements specify the testing plan's details.
<description>
High level description of the testing plan. The <description> element is followed by 1 or more <step> elements.
</description>
<step>
Detailed description of 1 step in the testing procedure. The content model of the <step> element is the same as the that of the <description> element (i.e., see section A Description of the <description> Element for details).
</step>
</plan>
</testing>
</component>


A Description of the <description> Element

This section describes the <description> element in detail.


Introduction

The <description> element contains text for human consumption. It allows you to organize your text into paragraphs, lists, and other structured content.

Note that the literal forms of the characters <, >, and & may not be used in element content. Instead you must type &lt;, &gt;, and &amp; respectively.


Common Usage

A <description> might contain only 1 or 2 sentences: 

<description>
  <p>My component is great.  It's so easy to use that no further
    information is needed. </p>
</description>

Note that even just 1 sentence must be enclosed in a paragraph element.

A <description> may contain only a few paragraphs: 

<description>
  <p>My component is great.  It's so easy to use that no further
     information is needed.</p>

  <p>I lied.  What follows is a description of my component.</p>
   .
   .
   .
</description>

You may refer to other material using the <cite>, <rlink>, or <slink> elements: 

<description>
  <p>
    My component does xyz.  It uses the xyz algorithm described in the paper
    <cite>The XYZ Algorithm</cite> by Xavier Yilmez Zideco.  A less formal
    discussion of the xyz algorithm is
    <slink path=ncrr/publications/xyz.html>available online</slink>. 
  </p>
</description>

The <cite> element marks text as the title of a paper or book (but is not itself a hyper-link).

The <slink> and <rlink> elements allow you to reference online material.

<Slink> links to material on the SCI web site (http://www.sci.utah.edu). It requires a path relative to http://www.sci.utah.edu. The path in the previous example resolves to http://www.sci.utah.edu/ncrr/publications/xyz.html.

<Rlink> links to material in the CVS tree (<rlink> actually links to material on the documentation web site but it is more easily explained as a path into the cvs tree). It requires a path relative to the location of your component specification document. For example, the path ./stuff.html resolves to the same directory where your component specification document lives.

A more general link element is not available because the documentation bureaucrats want some control over the material to which you link.

The <developer> element encloses material that is of interest only to developers: 

<description>
  .
  .
  .
  <developer>
    <p>Some smart developer ought to improve this rubbish.</p>
  </developer>
  .
  .
  .
<description>


Reference

Below, the description element's XML structure and corresponding logical content are described. Each element's start tag is followed by a description of its purpose and content, followed by its nested elements, and finally by its end tag. Basic nesting relationships among elements are shown by indentation.

Note: Additional features may be added to the <description> element in the future.

<description>
Starts a description. A description consists of any combination of paragraphs (<p>), lists (<orderedlist>, <unorderedlist>, and <desclist>), admonitions (<note>, <tip>, or <warning>), and developer notes (<developer>).
<p>
Starts a paragraph. A paragraph consists of character data, phrases (<term>, <keyword>, <keyboard>, <cite>, and <acronym>) and links (<slink> and <rlink>).
<term>
Encloses a term. A term is a word or short phrase you are introducing to your readers for the first time.
</term>
<keyword>
Encloses a keyword.
</keyword>
<keyboard>
Encloses material that is typed at a keyboard.
</keyboard>
<cite>
Encloses a reference to a paper or other publication. Is not itself a hyper-link but it may be used within the <slink> and <rlink> elements.
</cite>
<acronym>
Encloses an acronym (e.g. <acronym>XML</acronym>).
</acronym>
<slink>
Defines a SCI web site relative link. It requires 1 attribute which is the path of the target relative to http://www.sci.utah.edu. Text bracketed by the start and end tags will be presented as a link in online documents. The text may include phrase elements.
</slink>
<rlink>
Defines a SCIRun CVS relative link (<rlink> actually links to material on the documentation web site but it is more easily explained as a path into the cvs tree). It requires a path relative to the location of your component specification document. Text bracketed by the start and end tags will be presented as a link in online documents. The bracketed text may include phrase elements.
</rlink>
</p>
<orderedlist>
An ordered (numbered) list. Contains 1 or more <listitem> elements.
<listitem>
An item in an ordered list. A <listitem> may contain paragraphs and nested lists.
</listitem>
</orderedlist>
<unorderedlist>
An unordered list. Contains 1 or more <listitem> elements.
<listitem>
An item in an unordered list. A <listitem> may contain paragraphs and nested lists.
</listitem>
</unorderedlist>
<desclist>
A description list. Contains a list of terms or short phrases and their definitions or descriptions. Consists of 1 or more <desclistitem> elements.
<desclistitem>
Contains 1 <desclistterm> and 1 <desclistdef>.
<desclistterm>
A word or short phrase (not enclosed in <p> element).
</desclistterm>
<desclistdef>
Definition or description of the corresponding term. May consist of a mix of paragraph elements and list elements.
</desclistdef>
</desclistitem>
</desclist>
<note>
Calls attention to a piece of information. Aimed towards users of the component. Presumably the information within a <note> element will be rendered in a special way. A note consists of paragraphs and lists only.
</note>
<tip>
Contains an especially helpful piece of information. Aimed towards users of the component. Presumably the information within a <tip> element will be rendered in a special way. A tip consists of paragraphs and lists only.
</tip>
<warning>
Contains a warning (e.g., ``Don't press the red button!''). Aimed towards users of the component. Presumably the information within a <warning> element will be rendered in a special way. A warning consists of paragraphs and lists only.
</warning>
<developer>
Starts a developer section. Material here should be aimed towards fellow programmers. The <developer> element acts like the <description> element except that <developer> elements may not be nested.
</developer>
</description>

Editing the Spec

An ordinary text editor may be used to create the component specification document's content. However, it is easy to get lost in the noise of the XML syntax. Therefore, it is best to use an XML/DTD aware editor. This type of editor will help you construct a valid component specification document.

[X]Emacs is one such editor. It supports an editing environment called XML mode (which is really a derivative of PSGML mode). XML mode highlights XML syntax, indents nested elements and their content, and automatically inserts elements and attributes based on the position of the insertion point. It is still possible to create invalid documents using [x]emacs XML mode.

The following sections describe the use of [x]emacs XML mode.


XML Mode - Do You Have It?

Recent versions of [x]emacs come with XML mode installed. You can check if yours does by typing M-x xml-mode. You don't have XML mode if you get the message [No match] in return.

If you are a University of Utah Computer Science person then you may use the XML mode installed under /usr/local/contrib/mcole/psgml. You must add the following bit of lisp code to your .emacs file: 

(setq load-path (append '("/usr/local/contrib/mcole/psgml") load-path))

If you are not a UUCS person and you don't have XML mode you may get it online.


XML Mode - .emacs

The following lisp code should be inserted into your .emacs file: 

; Tell emacs to use sgml/xml mode for the following file types.
(setq auto-mode-alist
      (append
        '(("\\.sgm" . sgml-mode)
          ("\\.sgml" . sgml-mode)
          ("\\.xml" . xml-mode))
       auto-mode-alist))
(autoload 'sgml-mode "psgml" "Major mode to edit SGML files." t)
(autoload 'xml-mode "psgml" "Major mode to edit XML files." t) 

; Customize sgml/xml-mode default settings.
(add-hook 'sgml-mode-hook (lambda () (setq sgml-indent-data t)))

; Create some faces for use with sgml/xml mode.
; Change colors to suite your fancy.
(make-face 'sgml-start-tag-face) 
(set-face-foreground 'sgml-start-tag-face "MediumSeaGreen") 
(make-face 'sgml-end-tag-face) 
(set-face-foreground 'sgml-end-tag-face "SeaGreen") 
(make-face 'sgml-entity-face) 
(set-face-foreground 'sgml-entity-face "Red") 
(make-face 'sgml-doctype-face) 
(set-face-foreground 'sgml-doctype-face "firebrick") 
(make-face 'sgml-comment-face) 
(set-face-foreground 'sgml-comment-face "blue") 

; Use faces defined above.
(setq sgml-set-face t)
(setq sgml-markup-faces 
      '((comment   . sgml-comment-face) 
        (start-tag . sgml-start-tag-face) 
        (end-tag   . sgml-end-tag-face) 
        (doctype   . sgml-doctype-face) 
        (entity    . sgml-entity-face)))


XML Mode - Getting Started

To start a new component specification with [x]emacs do this:

  1. Use the C-x C-f (find-file) command to create a new file. The file's name must have the suffix .xml.
  2. Insert the preamble: 
    <?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE component SYSTEM 
    "<top-of-cvs-tree>/doc/ReferenceGuide/component.dtd">
    Replace <top-of-cvs-tree> with the path to the top of your SCIRun CVS tree.
  3. Select the Parse DTD item from the DTD menu which can be found in the window's main menu bar, or type C-c C-p.
  4. Insert the top level <component> element using [x]emacs pop-up menu (shift-button1 or shift-button3 for emacs and button3 for xemacs): Position the cursor at the end of the file and select the item named component from the pop-up menu. After doing this you will be prompted for the <component> element's name and category attributes. attribute. After typing in each of the attributes [X]Emacs will then insert a number of other required elements.
  5. Insert content into these elements. For example, you may insert your name between the start and end tags of the <author> element. Refer to section Rules, Structure, and Content if you have question about the content of any elements.
  6. The <gui> element is not automatically inserted by [x]emacs because a gui is optional. Most components will support a gui though so you will probably need to insert the <gui> element. Do this by positioning the cursor after the end tag of the <io> element. Then select the <gui> element from [x]emacs pop-up menu. [X]Emacs will insert the <gui> and its required sub-elements.
  7. Complete the component specification by adding content, elements, and attributes.

Note:

  • Most elements (like the <description> element) are composed of sub-elements. To add sub-elements, position the cursor between the start and end tags of the element and then insert an element using [x]emacs pop-up menu. The pop-up menu will list elements that are valid at the insertion point.
  • A few elements (namely <slink> and <rlink>) elements take attributes. If these elements are added via [x]emacs pop-up menu then [x]emacs will prompt you for the element's attribute values. If these element's are added manually then you may add the attributes via the pop-up menu: position the cursor anywhere within the element's start tag and select the desired attribute from the pop-up menu's list of attributes. [X]Emacs will prompt you for the value of the attribute and insert it in the element's start tag.


Validation

You must validate your component specification document to ensure that it is complete and correct.

A validation tool can be found online.

To use this tool you will need to change the second line of your document's preamble section to the following: 

<!DOCTYPE component SYSTEM 
  "http://www.sci.utah.edu/scirun_docs/doc/ReferenceGuide/component.dtd">

This change will break emacs' XML mode though so be sure to revert to the original before using emacs.
