CVS Repository Information
 

The Basics

Before you can use CVS, you have to let it know which server to use, what your login name is for that machine, what the path is to the repository you want, and finally what shell to use to access the server.

The server used for SCIRun is:

  csf.cs.utah.edu

and the repository is located in the following path:

  /csafe_noexport/cvs/cvsroot

To let CVS know which server and path to use, set the following environment variable:

  setenv CVSROOT USERNAME@SERVER:PATH

Which, for SCIRun (and username guest), would look like:

  setenv CVSROOT guest@csf.cs.utah.edu:/csafe_noexport/cvs/cvsroot

Now that CVS knows where to go, you need to let it know how to log on.  The SCIRun CVS server only allows secure shell access.  Set the following environment variable:

  setenv CVS_RSH COMMAND

Which, for SCIRun, would look like this:

  setenv CVS_RSH ssh

You should also set the following environment variable, to let the server know which file permissions should be assigned to files you add or modify:

  setenv CVSUMASK 000

Once you have all the environment variables set, you can use CVS to check-out code, add or modify code, and commit (or check-in) code.

Checking Out Code From the Repository

Checking out code more or less means downloading the whole source tree for the latest version of a software project.  To initiate the download you should type the following:

  cd PATH
  cvs checkout PROJECT

where PATH is the path to where you would like the root of the new source tree to be located, and PROJECT is the name of the project you wish to get.  For SCIRun the cvs command would look like:

  cvs checkout SCIRun
 

Adding a New File or Directory To the Repository

To add a new directory to the repository, type the following:

     cd PATH
  cvs add NEW_DIR

Where PATH is the path to the directory you want to add, and NEW_DIR is the name of the new directory.

To add a new non-directory file to the repository, type the following:

  cd PATH
  cvs add NEW_FILE
  cvs commit -m "initial commit of NEW_FILE"

Where PATH is the path to the file you want to add, and NEW_FILE is the name of the new file.  Note that to add a new file, it must already exist in the PATH that you've indicated.
 

Commiting Changes

Committing changes means that you have somehow modified code that you checked-out,  and (since it's way better now than it was) you want the changes to be made to the version in the repository.  Great care must be taken when committing changes to the repository.  If the code you commit is somehow dependent on a particular machine, environment, or platform, you will likely be responsible for what is known as "breaking the build" - a very bad thing.  A broken build occurs when code is committed to the repository without first verifying that the code _at least_ compiles and links on all the platforms that are required for that project.  If you commit code that doesn't compile and link on the automatic build machines, then you will be stripped naked, flogged in the center of the lab and hung upside down by your pinky toes for no less than a three week period - during which time people may freely fling undesirable things at your feeble nude body.

Once you have made changes to some files and you want to commit those changes, type the following:

  cd ROOT_PATH
  cvs update
  cvs commit -m "explain your changes"

Where ROOT_PATH is the root of the project's directory tree.  The above instructions will insure that ALL of the changes in your tree will also be made to the repository.  If you have some changes in your tree that you do not yet wish to commit, then you should commit on a file-by-file basis:

  cd PATH
  cvs update
  cvs commit -m "explain your changes" FILE_LIST

Where PATH is location and FILE_LIST is a list of the files you wish to commit.  Committing on a file-by-file basis needs to be done with caution.  If the code you commit relies on code that you chose not (or forgot) to commit, then you will "break the build"!
