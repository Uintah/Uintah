Overview

Chapter 1 discusses the general layout and core functionality of a standard PowerApp, and some design issues that arise when building such an application.

General Layout

The typical layout for a SCIRun PowerApp is shown in Figure 1. The menu consists of:

  • On the left, there is a Processing Pane that guides the user through the stages of the application.
  • In the middle, there is a Rendering window where all data interaction and visualization is performed.
  • On the right, there is a Visualization Pane where many parameters for the Rendering window can be controlled.

Figure 1: General PowerApp Layout

A smaller scale PowerApp may not need a Processing Pane and Visualization Pane. Either can be removed as shown in comparing the BioTensor and BioFEM apps.

Each PowerApp should have the standard menu as illustrated in Figure 2. The menu consists of:

  • File -> Save Session... - Saves a session for a specific PowerApp
  • File -> Load Session... - Loads a session for a specific PowerApp
  • File -> Quit - Quits the PowerApp
  • Help -> Show Tooltips - Turns tooltips on/off
  • Help -> Help Contents - References Tutorial and may show splashscreen
  • Help -> About - Displays a brief summary of PowerApp

Figure 2: PowerApp Menus

Core Functionality

To view the Rendering window at full-screen resolution, there must be support for detaching and reattaching the three panes. Do this by clicking on the vertical dashed lines between the panes. See detached panes in Figure 3.

Figure 3: Panes Detached from Viewer

At the bottom of the Processing Pane or Visualization Pane, there should be a progress indicator with a label and an indicator. The label indicates the current processing or visualization step. It can also indicate dynamic compilation. Below the indicator label is a progress indicator. When nothing is happening, the progress indicator should be in a reset state. When modules are executing, a spinning "candy cane" is displayed. When all executing modules are done, the complete state appears. In the case of an error, the progress indicator turns red. Click on the indicator to view the error module's log. The four indicator states are shown in Figure 4.

Design Issues

Some design issues related to PowerApps include interaction with SCIRun modules and their variables, and hidding complexities of dataflow from the user.

A PowerApp runs SCIRun in the background. In general, PowerApp widgets reference module variables. There should be a global array called mods. After loading the net, mods is set up to point to any modules that need to be referenced in the PowerApp. An example of setting the mods array is shown in Figure 5.

Figure 5: Setting the mods Array

Althouigh there is a dataflow network underneath, the user needs to see logical steps in execution. During initialization of a PowerApp, certain modules are disabled to prevent the user from moving on to a step beyond the first step. Upon executing a step, modules are enabled, allowing the user to move on.

In addition to disabling modules, widgets for future processing and visualization steps need to be disabled, and only enabled at the appropriate time. Processing tabs should have an button along with a button. These buttons can be enabled or disabled to help enforce the order of steps.

PowerApps support attaching and detaching panes from the Viewer. A developer must make sure the attached and detached panes are in sync. When a tab is selected, the corresonding tab must also be selected in the other pane. When widgets are disables with a radiobutton or checkbutton, the same widgets must be disabled in the other pane. It is usefull to have methods that keep these states in sync, especially when handling the loading of a session.

Summary

Chapter 1 discussed the general layout and functionality of a PowerApp, and issues that arise when building a PowerApp. Chapter 2 discusses the anatomy of an .app file.
